# Question

Why does OT-Mix sometimes do worse than baseline despite finding meaningful hard negatives? Is there a per-step signal that can gate OT pressure to prevent destabilization?

---

# Current Understanding

Three CUB-200 runs through ep22–25 (baseline complete, adaptive complete, mixed running):

| Ep | Baseline | Adaptive | Mixed |
|---|---|---|---|
| 10 | 0.41% | 0.47% | 0.45% |
| 15 | 0.58% | 0.54% | **0.68%** |
| 20 | 0.78% | 0.85% | **0.89%** |
| 21 | 0.87% | — | **0.98%** |
| 50 | **1.38%** | 1.28% | ? |

Adaptive (ep50): best 1.35% at ep49, final 1.28%. Baseline finishes 0.03–0.10pp ahead. Trajectory non-monotone in all three runs.

OT finds real structure: from ep7 onward, selected_neg_rank ≈ 1–2 (same-species), coupling_entropy 2.2–2.5. The failure is not that OT cannot find hard negatives. The failure is that it applies hard-negative pressure unconditionally regardless of per-step plan quality.

---

# Gap

Three identified failure regimes in the logs:

**A. Diffuse plan:** entropy > 3.0, rank 30+. Examples: ep1 step 92 (entropy 3.17, rank 33.5), ep2 step 100 (entropy 3.43, rank 31.7). OT plan is near-uniform; synthetics are random interpolations. OT contributes noise.

**B. Stale plan / too easy:** plan sharp, but selected rank 30+ and pos_selected_gap > +0.10. This happens every end-of-epoch (plan cached, model has moved). The positive is already well clear of the synthetic. OT wastes the loss budget.

**C. Synthetic too dominant:** plan sharp, rank 1–2, but pos_selected_gap < −0.07. Synthetic is beating the positive. OT adds hard-negative pressure before base alignment has stabilized. Creates high-variance updates, consistent with the observed ep21→ep22 dip in mixed (0.98%→0.69%) and ep49→ep50 in adaptive (1.35%→1.28%).

No per-step mechanism existed to distinguish these from the useful regime (-0.05 ≤ gap ≤ 0.05, entropy < 3.0, low rank).

---

# Hypothesis

> OT-Mix instability comes less from synthetic negatives themselves and more from applying the OT loss without conditioning on per-step signal quality. The useful regime is when the transport plan is sharp and the selected synthetic is a near-boundary negative. When the plan is diffuse, the selected rank is random, or the synthetic already beats the positive by a large margin, OT becomes noise or unstable hard-negative pressure. A conditional `alpha_effective` based on coupling entropy, selected rank, and `Pos - Selected Gap` should reduce oscillation while preserving the benefit of structured hard negatives.

This is falsifiable:
> If OT-Mix instability comes from unconditional OT pressure, gated alpha should reduce validation oscillation and improve or preserve best R@1.

---

# Implementation

**New:** `compute_alpha_effective()` pure function in `model/loss.py`:

```
if alpha == 0:              → (0.0, bucket=4 inactive)
if entropy > 3.0:           → (0.0, bucket=3 diffuse)
if gap > 0.10:              → (0.0, bucket=1 too_easy)
if gap < -0.07:             → (alpha * 0.25, bucket=2 too_hard)
else:                       → (alpha, bucket=0 useful)
```

Thresholds are constructor params (`gap_suppress_easy=0.10`, `gap_downweight_hard=-0.07`, `hard_alpha_scale=0.25`). Initial values — do not tune until firing distribution is observed.

**New loss_dict keys (all numeric):** `alpha_scheduled`, `alpha_effective`, `ot_suppressed_entropy`, `ot_suppressed_too_easy`, `ot_downweighted_too_hard`, `gap_bucket_id`, `gap_bucket_useful`, `gap_bucket_too_easy`, `gap_bucket_too_hard`.

**Unit tests:** 8 tests in `tests/test_alpha_gating.py` — all 4 branches, inactive case, entropy-takes-priority, both boundary conditions. All pass.

**New experiment:** `cub200_softmax_mix_adaptive_gated` — identical to adaptive (seed 42, same OT config), conditional alpha active.

# Analysis Framework

**Live-reading checklist during training:**
- Is `alpha_effective` often 0 during stale-plan / too-easy steps?
- Are useful-bucket steps still happening often enough?
- Does `too_hard` dominate during mid-training (ep7–20)?
- Does entropy suppression mostly happen early (ep1–4) or throughout?
- Do validation dips correlate with high `too_hard` frequency that epoch?

**Decision rules:**

| Outcome | Interpretation |
|---|---|
| R@1 improves + std decreases | Gating fixed instability — strong result |
| R@1 same + std decreases | Gating stabilizes but OT signal still weak |
| R@1 worse + alpha mostly suppressed | Too conservative; easy/stale steps had value |
| R@1 worse + alpha mostly active | Gating idea not addressing real failure mode |
| `too_hard` dominates | Synthetics overpower positives; lower alpha or hard_alpha_scale |
| `too_easy` dominates | OT selection often non-informative; reconsider update schedule |
| Entropy suppression dominates late | Plan quality unstable; update frequency is next target |

# Next Question

After gated run completes: if `too_easy` dominates (end-of-epoch stale steps are majority of suppressions), the real fix may be increasing `update_freq` so the plan is fresher, rather than suppressing those steps. If `too_hard` dominates mid-training, consider a softer `hard_alpha_scale` (0.1 instead of 0.25).
